# MongoDB Sharding with Replication - Задание 3

Проект реализует шардирование MongoDB с двумя шардами и репликацией для повышения производительности и отказоустойчивости онлайн-магазина "Мобильный мир".

## Архитектура

### Config Server (3 реплики)
- **configSrv** - первая реплика config server для хранения метаданных о шардах
- **configSrv2** - вторая реплика config server
- **configSrv3** - третья реплика config server

### Shard 1 (3 реплики)
- **shard1-1** - первая реплика шарда 1
- **shard1-2** - вторая реплика шарда 1
- **shard1-3** - третья реплика шарда 1

### Shard 2 (3 реплики)
- **shard2-1** - первая реплика шарда 2
- **shard2-2** - вторая реплика шарда 2
- **shard2-3** - третья реплика шарда 2

### Инфраструктура
- **Mongos Router** (`mongos_router`) - роутер для распределения запросов между шардами
- **Приложение** (`pymongo_api`) - FastAPI приложение для работы с БД

### Особенности архитектуры

**Отказоустойчивость:**
- Каждый шард имеет 3 реплики, что позволяет продолжить работу при выходе из строя до 1 ноды в каждом шарде
- Config server также имеет 3 реплики для обеспечения отказоустойчивости метаданных кластера
- Автоматическое переключение на резервные реплики (failover) в случае сбоя

**Производительность:**
- Данные распределены между двумя шардами для горизонтального масштабирования
- Возможность чтения из вторичных реплик для распределения нагрузки

## Запуск проекта

### 1. Запустить все сервисы

```bash
docker compose up -d --build
```

### 2. Дождаться запуска MongoDB контейнеров

Проверить статус контейнеров:

```bash
docker compose ps
```

Должны быть в статусе `healthy` все 3 контейнера Config Server (`configSrv`, `configSrv2`, `configSrv3`). Это может занять 10-15 секунд.

**Примечание:** Контейнеры шардов (`shard1-*`, `shard2-*`), `mongos_router` и `pymongo_api` будут в статусе `unhealthy` до выполнения инициализации (шаг 3), так как они настроены как replica sets, которые требуют инициализации.

### 3. Инициализировать шардирование и репликацию

Выполнить скрипт инициализации:

```bash
bash mongo-init.sh
```

Скрипт автоматически выполнит 14 шагов:
1. Инициализацию replica set для config server (3 члена)
2. Инициализацию replica set для shard1 (3 члена)
3. Инициализацию replica set для shard2 (3 члена)
4. Добавление shard1 в кластер с указанием всех реплик
5. Добавление shard2 в кластер с указанием всех реплик
6. Включение шардирования для базы данных `somedb` и коллекции `helloDoc`
7. Наполнение коллекции 1000 документами
8. Проверку метаданных шардирования в mongos router (5 метрик)
9. Проверку распределения документов в shard1
10. Проверку распределения документов в shard2
11. Проверку репликации для config server
12. Проверку репликации для shard1
13. Проверку репликации для shard2
14. Финальную проверку, что все 11 контейнеров теперь `healthy`

После успешного выполнения скрипта все контейнеры (включая шарды, mongos_router и pymongo_api) перейдут в статус `healthy`.

### 4. Проверить работу

Открыть в браузере или выполнить curl:

```bash
curl http://localhost:8080/
```

Ответ должен содержать:
- `"mongo_topology_type": "Sharded"` - подтверждение работы шардирования
- `"shards"` с информацией о двух шардах
- `"collections"` с количеством документов (≥ 1000)
- Информацию о репликах для каждого шарда (по 3 реплики на шард)

### Проверка репликации

Вы можете проверить статус репликации для каждого шарда:

**Для config server:**
```bash
docker compose exec configSrv mongosh --port 27017 --quiet --eval "rs.status()"
```

**Для shard1:**
```bash
docker compose exec shard1-1 mongosh --port 27018 --quiet --eval "rs.status()"
```

**Для shard2:**
```bash
docker compose exec shard2-1 mongosh --port 27019 --quiet --eval "rs.status()"
```

В выводе каждой команды вы должны увидеть 3 члена replica set.

## Запуск тестов

```bash
docker compose --profile test up --build pymongo_api_test
```

Тесты автоматически дождутся, пока все сервисы станут `healthy` перед запуском.

## Остановка проекта

```bash
docker compose down
```

Для удаления данных:

```bash
docker compose down -v
```

## Проверка отказоустойчивости

Вы можете проверить отказоустойчивость, остановив одну из реплик:

```bash
docker compose stop shard1-2
```

После этого приложение должно продолжить работу, используя оставшиеся реплики shard1-1 и shard1-3.

Для восстановления реплики:

```bash
docker compose start shard1-2
```

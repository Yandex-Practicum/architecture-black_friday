# Выбор метода кеширования

## Требования к кешированию

В условиях высоконагруженной системы для распродажи Black Friday кеширование необходимо для:

1. **Снижение нагрузки на MongoDB**
   - Уменьшение количества запросов к базе данных
   - Разгрузка шардов MongoDB при пиковых нагрузках
   - Снижение утилизации CPU/RAM на серверах БД

2. **Ускорение доступа к часто запрашиваемым данным**
   - Быстрый доступ к данным из оперативной памяти
   - Кеширование редко изменяемой информации (каталог товаров, категории)
   - Повышение скорости отклика API для пользователей

3. **Отказоустойчивость системы кеширования**
   - Работоспособность при отказе отдельных нод кеша
   - Автоматическое восстановление после сбоев
   - Репликация данных для предотвращения потери кеша

4. **Управляемость и согласованность данных**
   - Механизмы инвалидации (сброса) кеша при обновлении данных
   - Контроль времени жизни (TTL) закешированных данных
   - Мониторинг эффективности кеширования (cache hit rate)

## Выбранный метод кеширования

**Redis Master-Replica**

### Описание метода

Redis Master-Replica - это конфигурация Redis с одним главным узлом (master) для записи и несколькими узлами-репликами (replica/slave) для чтения.

Решение состоит из:
- **1 Master** - принимает все операции записи (SET, DEL)
- **2 Replica** - асинхронно реплицируют данные от master, обслуживают запросы на чтение (GET)
- **Redis Sentinel** - мониторит master и автоматически выполняет failover при отказе

Механизм работы:
1. Приложение записывает данные в master
2. Master асинхронно реплицирует данные на replica
3. Приложение читает данные с master или replica (балансировка на уровне драйвера)
4. При отказе master одна из replica становится новым master

### Обоснование выбора

Для проекта Black Friday Redis Master-Replica является оптимальным выбором:

1. **Простота внедрения и поддержки** - меньше узлов чем в Redis Cluster, проще настройка
2. **Отказоустойчивость** - при отказе master автоматический failover на replica
3. **Масштабирование чтения** - 2 replica обслуживают запросы на чтение, снижая нагрузку на master
4. **Консистентность с архитектурой MongoDB** - аналогичный подход к репликации
5. **Достаточная производительность** - для одного приложения хватает одного master

### Применение в проекте

**Что кешируется:**
- **Каталог товаров** - данные редко меняются, часто запрашиваются
- **Информация о пользователях** - данные редко меняются, часто запрашиваются
- **Категории и фильтры** - статичные данные для фильтрации товаров
- **Результаты агрегаций** - сложные запросы с COUNT/SUM по MongoDB

**Примерные параметры TTL:**
- Каталог товаров: 10 минут (может измениться при обновлении акций)
- Пользователи: 5 минут
- Категории: 1 час (почти статичны)

## Альтернативные варианты кеширования

### Redis Cluster (Distributed Cache)

**Описание:** Распределенная система кеширования с шардированием данных между несколькими master-нодами, каждая со своими репликами.

**Минимальная конфигурация:**
- 3 master-ноды (для кворума)
- 3 replica-ноды (по 1 на каждый master)
- Итого: 6 нод Redis

**Преимущества:**
- Горизонтальное масштабирование памяти (данные распределены по шардам)
- Высокая доступность (failover для каждого шарда)
- Высокая производительность при очень больших объемах данных

**Почему не подходит:**
- **Избыточная сложность** - для текущего проекта достаточно одного master с репликами
- **Overhead на координацию** - протокол gossip для синхронизации состояния кластера
- **Увеличение ресурсов** - минимум 6 нод вместо 3
- **Сложность отладки** - распределенные данные сложнее диагностировать

**Допустимое применение:** В перспективе может быть целесообразно, если объем кешируемых данных не поместится на одном master

### Memcached

**Описание:** Простая in-memory система кеширования типа key-value без встроенной репликации.

**Преимущества:**
- Простота использования (минимальная конфигурация)
- Очень высокая производительность за счет минимализма
- Меньшее потребление памяти на хранение данных

**Почему не подходит:**
- **Отсутствие репликации** - нет встроенной отказоустойчивости, риск поймать перезапуск во время распродажи и "положить" БД запросами
- **Нет персистентности** - при перезапуске весь кеш теряется, риск поймать перезапуск во время распродажи и "положить" БД запросами
- **Ограниченные типы данных** - только строки, нет структур (lists, sets, hashes)
- **Нет TTL на уровне ключей** - используется LRU eviction

### Application-level Caching (in-memory в приложении)

**Описание:** Кеширование данных непосредственно в памяти приложения (например, словарь Python, Map в Node.js).

**Преимущества:**
- Нулевая задержка сети (данные в процессе приложения)
- Не требует отдельной инфраструктуры
- Простота реализации

**Почему не подходит:**
- **Не переживает перезапуск** - при деплое новой версии кеш теряется
- **Не масштабируется горизонтально** - каждый инстанс приложения имеет свой кеш
- **Ограничение памяти** - конкурирует с памятью приложения
- **Сложность инвалидации** - при обновлении данных нужно сбросить кеш на всех инстансах

## Недостатки, ограничения, риски

### 1. Проблема согласованности данных (Cache Coherence)

**Описание риска:**
- Данные в Redis могут устареть при обновлении в MongoDB
- Пользователь может видеть старую информацию после изменения данных
- Особенно критично для цен, количества товара на складе, статусов заказов

**Меры по минимизации:**
- **Явная инвалидация кеша** при обновлении данных в MongoDB (паттерн Cache-Aside)
- **Короткие TTL** для критичных данных (цены: 1-2 минуты)
- **Избирательное кеширование** - не кешировать часто изменяемые данные
- **Мониторинг актуальности** - логирование промахов кеша для выявления проблем

### 2. Задержка репликации (Replication Lag)

**Описание риска:**
- Replica отстают от master на несколько миллисекунд или секунд
- Пользователь может прочитать старые данные с replica сразу после записи в master
- Проблема "read-after-write consistency"

**Меры по минимизации:**
- **Чтение с master** для критичных сценариев (чтение сразу после записи)
- **Настройка read preference** в драйвере Redis
- **Мониторинг replication lag** - алерты при задержке > 1 секунды
- **Использование Redis Sentinel** для контроля состояния репликации

### 3. Простой при отказе master

**Описание риска:**
- При отказе master запись в кеш временно невозможна
- Во время failover (переключения replica на master) возможен простой
- Потенциальная потеря данных, записанных в master, но еще не реплицированных

**Меры по минимизации:**
- **Использование Redis Sentinel** - автоматический failover
- **Graceful degradation** - при недоступности Redis работать напрямую с MongoDB
- **Мониторинг доступности** master с автоматическими алертами
- **Настройка min-replicas-to-write** - master принимает записи только если N реплик доступны
- **Настройка Sentinel кворума** - например 2 из 3 Sentinel должны согласиться на failover

### 4. Ограничение памяти

**Описание риска:**
- Redis хранит все данные в RAM - при переполнении памяти начинается eviction
- Eviction политика (например, LRU) может удалить важные данные
- Стоимость RAM выше стоимости дискового пространства

**Меры по минимизации:**
- **Расчет необходимой памяти** перед внедрением (размер объектов × количество)
- **Настройка maxmemory-policy** - выбор стратегии eviction (allkeys-lru, volatile-ttl)
- **Мониторинг используемой памяти** - алерты при заполнении > 80%
- **Селективное кеширование** - кешировать только часто используемые данные
- **Сжатие данных** - использование эффективных форматов (MessagePack вместо JSON)

### 5. Холодный старт кеша (Cold Start)

**Описание риска:**
- После перезапуска Redis или деплоя кеш пуст
- Все запросы идут в MongoDB, создавая всплеск нагрузки
- Производительность системы деградирует до заполнения кеша

**Меры по минимизации:**
- **Redis persistence (RDB/AOF)** - сохранение данных на диск для восстановления
- **Cache warming** - предварительное заполнение кеша при старте
- **Постепенное включение** кеша после старта (процент запросов)
- **Rate limiting** на MongoDB для защиты от всплеска нагрузки

### 6. Сложность debugging и мониторинга

**Описание риска:**
- Сложно отследить, откуда пришли данные (кеш или БД)
- Трудно диагностировать проблемы с кешем (устаревшие данные)
- Необходимость мониторить дополнительные метрики (hit rate, latency)

**Меры по минимизации:**
- **Логирование источника данных** - промахи и попадания в кеш
- **Метрики кеширования:**
  - попадания в кеш
  - промахи кеша
  - среднее время отклика из кеша
- **Система мониторинга** для визуализации эффективности кеша (например, Prometheus + Grafana)

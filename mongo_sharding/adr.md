# Выбор метода шардирования

## Требования к шардированию

В условиях высоконагруженной системы для распродажи Black Friday необходимо обеспечить:

1. **Преодоление технических ограничений**
   - Обработка большого объёма данных (заказы, товары, пользователи)
   - Потенциал для дальнейшего роста при увеличении нагрузки

2. **Повышение надёжности системы**
   - Отказ одного сервера не должен приводить к полной остановке работы
   - Обеспечение доступности данных даже при сбоях отдельных компонентов

3. **Ускорение доступа к данным**
   - Быстрая обработка простых запросов (получение информации о товаре, пользователе)
   - Распределение нагрузки между серверами для снижения конкуренции запросов

4. **Равномерное распределение нагрузки**
   - Предотвращение "горячих точек" (hot spots) при всплесках активности
   - Балансировка операций чтения и записи между шардами

## Выбранный метод шардирования

**Хэшированное шардирование**

### Описание метода

Хэшированное шардирование - это разделение данных на шарды на основе хэш-функции. Хэш-функция принимает входные данные (например, ID пользователя или ID товара) и возвращает хэш-значение, которое определяет, в какой шард попадёт каждая запись данных.

Алгоритм обладает следующими свойствами:
- Детерминированность: применение к одному и тому же значению всегда даёт одинаковый результат
- Необратимость: невозможно восстановить исходное значение из хэша
- Равномерность распределения: данные распределяются по шардам равномерно

### Обоснование выбора

Для проекта Black Friday хэшированное шардирование является оптимальным выбором по следующим причинам:

1. **Равномерное распределение данных** - критично важно для обработки всплесков нагрузки во время распродажи
2. **Высокая производительность** - быстрое определение шарда без обращения к дополнительным сервисам
3. **Отсутствие единой точки отказа** - каждый шард работает независимо
4. **Простота масштабирования** - легко добавлять новые шарды при росте нагрузки

### Применение в проекте

- **Шардирование пользователей**: по user_id с использованием хэш-функции
- **Шардирование заказов**: по order_id для равномерного распределения
- **Шардирование товаров**: по product_id для балансировки каталога

## Альтернативные варианты шардирования

### Диапазонное шардирование (Range Based Sharding)

**Описание:** Разделение данных на основе диапазона значений. Например, товары стоимостью до 1000₽ - в первый шард, от 1000₽ до 5000₽ - во второй, свыше 5000₽ - в третий.

**Преимущества:**
- Лёгкая реализация
- Быстрый поиск информации по диапазону
- Простота выполнения целевых операций (например, запросы по ценовым категориям)

**Почему не подходит:**
- **Риск дисбаланса**: во время Black Friday товары определённых ценовых категорий могут быть более популярны, что приведёт к неравномерной нагрузке на шарды
- **"Горячие точки"**: шард с популярными товарами будет перегружен, в то время как другие недогружены
- **Непредсказуемость нагрузки**: сложно предугадать, какой диапазон будет наиболее востребован

### Динамическое шардирование (Dynamic Sharding)

**Описание:** Автоматическое масштабирование хранилища в зависимости от текущей производительности и объёма данных. Использует внешний поисковый сервис для определения местоположения записей.

**Преимущества:**
- Гибкость и адаптивность
- Автоматическая балансировка при изменении нагрузки
- Возможность перемещения данных между шардами

**Почему не подходит:**
- **Высокая сложность реализации**: требует сложной балансировки нагрузки и тщательно продуманной архитектуры
- **Риск единой точки отказа**: внешний поисковый сервис становится критичным компонентом
- **Временные затраты**: сложно реализовать в краткосрочной перспективе
- **Непредсказуемость**: во время пиковых нагрузок перемещение данных может снизить производительность

### Геошардинг (Geo Sharding)

**Описание:** Хранение в разных сегментах информации, относящейся к определённой географической зоне.

**Преимущества:**
- Низкая задержка для пользователей из определённых регионов
- Соблюдение требований по локализации данных
- Оптимизация для мобильных приложений с учётом геолокации

**Почему не подходит как основной метод:**
- **Неравномерность нагрузки**: разные регионы имеют разную активность во время распродажи
- **Сложность балансировки**: сложно предсказать нагрузку по регионам
- **Ограниченная применимость**: подходит для данных с явной географической привязкой, но не для всех типов данных в системе

Геошардинг может быть использован в перспективе в комбинации с хэшированным шардированием для оптимизации доступа к данным в разных регионах.

## Недостатки, ограничения, риски

### 1. Сложность реализации

**Описание риска:**
- Реализация шардирования требует значительных усилий и сопряжена с риском ошибок при разделении данных
- Неправильная конфигурация может привести к потере информации или повреждению хранилища
- Необходимость изменения логики приложения для работы с распределёнными данными

**Меры по минимизации:**
- Использование проверенных решений (MongoDB Sharding, Redis Cluster)
- Тщательное (в идеале - автоматизированное, с журналированием покрытия) тестирование на тестовых средах перед развёртыванием
- Поэтапное внедрение с постепенным переносом данных
- Создание детальной документации по архитектуре и конфигурации

### 2. Риск снижения эффективности разработки

**Описание риска:**
- Управление данными из нескольких сегментов вместо единой точки входа усложняет разработку
- Расширение схемы БД становится сложнее
- Расследование проблем требует анализа данных из множества шардов
- Необходимость поддерживать механизм шардирования отвлекает ресурсы от разработки функциональности

**Меры по минимизации:**
- Использование ORM и библиотек, поддерживающих работу с шардированными БД
- Создание абстракций для работы с данными, скрывающих детали шардирования
- Разработка инструментов для мониторинга и диагностики всех шардов
- Автоматизация операций по управлению шардами

### 3. Неравномерность загрузки серверов

**Описание риска:**
- Несмотря на использование хэш-функции, возможна несбалансированность данных
- При добавлении или удалении шардов требуется перебалансировка данных
- Повторное сегментирование может вызвать простой серверов

**Меры по минимизации:**
- Выбор качественной хэш-функции с хорошим распределением (например, MD5, MurmurHash)
- Использование согласованного хеширования для минимизации перемещения данных при изменении количества шардов
- Мониторинг загрузки шардов и своевременная перебалансировка
- Планирование достаточного количества шардов с учётом будущего роста

### 4. Снижение скорости обработки сложных запросов

**Описание риска:**
- Запросы, требующие обращения к нескольким шардам одновременно (JOIN, агрегация), выполняются медленнее
- Транзакции, охватывающие несколько шардов, становятся сложными
- Потеря производительности по сравнению с монолитной базой при сложных выборках

**Меры по минимизации:**
- Денормализация данных для минимизации межшардовых запросов
- Кэширование результатов сложных запросов (Redis)
- Использование паттерна CQRS для разделения операций чтения и записи
- Оптимизация схемы данных с учётом шардирования (размещение связанных данных в одном шарде)
- Ограничение использования транзакций, охватывающих несколько шардов

### 5. Сложность добавления новых шардов

**Описание риска:**
- При использовании простой хэш-функции добавление нового шарда требует перераспределения большого объёма данных
- Во время миграции данных возможно снижение производительности

**Меры по минимизации:**
- Использование согласованного хеширования для минимизации объёма перемещаемых данных
- Выполнение миграции в периоды низкой нагрузки
- Постепенная миграция с контролем нагрузки на систему
- Резервирование достаточного количества шардов "на будущее"

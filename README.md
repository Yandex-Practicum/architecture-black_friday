## Задание 5. Service Discovery и балансировка (API Gateway)

В четвёртом варианте архитектуры реализовано горизонтальное масштабирование приложения.

### Ключевые элементы:

* **API Gateway** — единая точка входа для клиентов, выполняет балансировку нагрузки между инстансами приложения.
* **Несколько инстансов приложения (`pymongo_api`)** — обеспечивают отказоустойчивость и масштабируемость.
* **Consul (Service Discovery)** — хранит информацию о доступных инстансах приложения и их состоянии.

### диаграмма ([тут](docs/diagrams/sprint_4_task_5_v_4.drawio))

### Принцип работы:

1. Клиент отправляет HTTP-запрос в API Gateway.
2. API Gateway обращается к Consul для получения актуального списка инстансов приложения.
3. Запрос балансируется между доступными инстансами.
4. Приложения используют Redis для кеширования и MongoDB (sharded + replicated) для хранения данных.

## Задание 6. CDN

В пятом варианте архитектуры добавлен **CDN** для ускорения доставки статического контента пользователям из разных регионов.

### Ключевые элементы:

* **CDN Edge-серверы** в нескольких регионах (Europe, Asia, USA).
* **Static Content Origin** — источник статического контента онлайн-магазина.

### Принцип работы:

1. Пользователь получает статический контент с ближайшего CDN Edge-сервера.
2. При отсутствии контента в кеше CDN запрашивает его у Origin-сервиса.
3. Динамические API-запросы (каталог, пользователи, данные) направляются напрямую в API Gateway, минуя CDN.

### диаграмма ([тут](docs/diagrams/sprint_4_task_6_v_5.drawio))

### Результат:

* Снижение latency для пользователей из разных регионов.
* Снижение нагрузки на backend-приложение.
* Повышение общей производительности и масштабируемости системы.
